## Context

APE (ICSE 2019) demonstrates superior coverage metrics over existing tools (64.11% activity coverage, 28.38% method coverage) but has a broken build system: Java 25 + Android build-tools 35 removed `dx` and reject `--source/--target 1.7`. The rv-android ecosystem already integrates the **original APE** as a built-in tool (`rv-tools/builtin/ape/tool.py`), but without any improvements.

**Research goal**: Create `ape-rv` — an enhanced APE with:
1. A working build (Maven + d8 + Java 11)
2. Improved UI coverage: ViewPager/tab navigation, OptionsMenu systematic exploration
3. MOP/RV awareness: static analysis integration for spec-guided exploration (optional)
4. An `aperv` plugin for rv-tools enabling controlled comparative experiments

**Comparison baseline**: original `ape` builtin vs `aperv` (enhanced) within the rv-experiment framework.

### Phase 1 — Build Migration
**SDD Track**: Quick Path
**Repository**: `workspace-rv/ape/`

Fix the broken build by replacing Ant+dx with Maven+d8+Java 11.

**Tasks**:
1. Create `pom.xml` with:
   - `javac --release 11`
   - `exec-maven-plugin` to invoke `d8` post-compile with `--output ape-rv.jar --min-api 23`
   - `framework/classes-full-debug.jar` and `dalvik_stub/classes.jar` as compile-time dependencies (scope: system or provided)
   - Output: `ape-rv.jar` containing `classes.dex`
2. Add `maven-resources-plugin` to copy `ape-rv.jar` on `install` to:
   `${aperv_tool_dir}/src/aperv_tool/tools/aperv/ape-rv.jar`
   (same pattern as rvsmart: `rvsec/rvsec-android/rvsmart/pom.xml` → `rv-android/modules/rvsmart-tool/src/rvsmart_tool/tools/rvsmart/rvsmart.jar`)
3. Deprecate `build.xml` (keep with notice)
4. Update `CLAUDE.md` build section (`ant` → `mvn`)
5. Update `openspec/config.yaml` language/build metadata

**Note on compile-time JARs**: `framework/classes-full-debug.jar` and `dalvik_stub/classes.jar` are **required** at compile time — APE uses `@hide` Android APIs (`android.app.UiAutomationConnection`, `android.hardware.display.DisplayManagerGlobal`, `com.android.internal.*`) that are absent from the public `android.jar`. Keep both JARs. Do NOT replace with `android-29.jar`.

**Critical files**:
- `build.xml` — deprecate
- `pom.xml` — create
- `src/com/android/commands/monkey/ApeAPIAdapter.java` — should not need changes (reflection-based)

**Verification**:

### Phase 2 — UI Enhancements
**SDD Track**: Fast-Forward
**Repository**: `workspace-rv/ape/`

> **Note (updated 2026-03-11 after deep code analysis):** Scope revised based on actual APE code inspection.

#### 2a. ~~BACK Action Fix~~ — REMOVED FROM SCOPE
APE's BACK action is already well-implemented:
- `ActionType.MODEL_BACK` exists and every `State` has a `backAction` (State.java:62)
- `SataAgent.selectNewActionEpsilonGreedyRandomly()` selects BACK when unvisited (highest priority)
- `generateKeyBackEvent()` → `generateKeyEvent(KeyEvent.KEYCODE_BACK)` executes it correctly
- The "BUG-01: BACK never executes" is a **rvsmart** bug, NOT an APE bug

#### 2b. ViewPager / Tab Support — CLASS NAME UPDATE ONLY
- **Problem**: APE detects `android.support.v4.view.ViewPager` (support library, pre-2018) but NOT modern AndroidX equivalents
- **Fix**: Update `GUITreeNode.getScrollType()` to add modern class names (1 file, ~2 lines)
- `"androidx.viewpager.widget.ViewPager"` → horizontal
- `"androidx.viewpager2.widget.ViewPager2"` → horizontal
- **Note**: Do NOT add `androidx.recyclerview.widget.RecyclerView` — RecyclerView can be vertical or horizontal; adding it unconditionally causes false positives on vertical lists
- `MODEL_SCROLL_LEFT_RIGHT` / `MODEL_SCROLL_RIGHT_LEFT` already exist and already generate correct swipe events
- **Files**: `ape/tree/GUITreeNode.java` only (line 506 area)

#### 2c. OptionsMenu Systematic Exploration — NEW MODEL_MENU ACTION
- **Problem**: `KeyEvent.KEYCODE_MENU` exists in fuzzing code but NOT as a systematic model action. Menu is only pressed randomly during fuzzing, not tracked in the state model
- **Fix**: Add `MODEL_MENU` to `ActionType` following the exact same pattern as `MODEL_BACK` (no target required, present in every state)

**All touch points** (5 files):
  1. `ActionType.java`: add `MODEL_MENU` to enum (after `MODEL_BACK`); verify `requireTarget()` range — `MODEL_MENU` must return `false` (like `MODEL_BACK`); `isModelAction()` must return `true`
  2. `State.java`: add `menuAction = new ModelAction(this, ActionType.MODEL_MENU)` in constructor; add `getMenuAction()` accessor
  3. `MonkeySourceApe.java`: add `case MODEL_MENU` in `generateEventsForActionInternal()` → call `generateKeyMenuEvent()`; add `MODEL_MENU` to `validateResolvedAction()` as always-valid (like `MODEL_BACK`)
  4. `GUITreeNode.java`: add `MODEL_MENU` to the explicit block list in `resetActions()` (prevents widget from receiving a non-widget action type)
  5. `SataAgent.java`: consider MENU unvisited with high priority after BACK check (same logic as BACK unvisited check)
  - After MENU press, the new GUITree captures menu items as regular clickable widgets — no special handling needed

**Files**: `ActionType.java`, `State.java`, `MonkeySourceApe.java`, `GUITreeNode.java`, `SataAgent.java`

**Verification**: Run on the 7 worst-performing APKs from prior experiments (those using ViewPager and OptionsMenu) and compare activity coverage against original APE.

### Phase 3 — MOP/RV Awareness (Optional)
**SDD Track**: Full SDD
**Repository**: `workspace-rv/ape/`
**Prerequisite**: static analysis JSON file for the target APK (produced by rv-android pipeline)

> **This phase is optional.** The `aperv:sata` variant runs without MOP data. The `aperv:sata_mop` variant activates MOP-guided scoring when `ape.mopDataPath` is provided.

> **Updated 2026-03-11 after deep code analysis.** Concrete integration points confirmed.

#### 3a. Static Analysis Data Loader — `MopData.java`

**JSON format** (produced by rv-android static analysis pipeline):
```json
{
  "reachability": [{ "className": "...", "methods": [{ "name": "...", "signature": "...", "directlyReachesMop": true, "reachesMop": true }] }],
  "windows": [{ "name": "com.example.app.MainActivity", "widgets": [
    { "idName": "btn_encrypt", "type": "Button", "listeners": [{ "handler": "<com.example.app.MainActivity: void onClick(android.view.View)>" }] }
  ]}],
  "transitions": [...]
}
```

**Widget → MOP mapping requires cross-reference** between `windows[]` and `reachability[]`:
- `windows[].widgets[].listeners[].handler` → handler method signature
- `reachability[].methods[].signature` + `.directlyReachesMop` / `.reachesMop` → MOP flags
- Join key: handler signature matches `reachability` method signature

**APE implementation**:
```java
// New: ape/utils/MopData.java
class MopData {
    // Map: activityClassName → (shortResourceId → {directMop, transitiveMop})
    Map<String, Map<String, WidgetMopData>> widgetData;
    Map<String, Boolean> activityHasMop;   // activity → has any MOP method

    static MopData load(String path);      // parse windows[] + reachability[], cross-reference by handler signature
    WidgetMopData getWidget(String activity, String shortId);
    boolean activityHasMop(String activity);
}
```

**Config flag** — add to `Config.java`:
```java
public static final String mopDataPath = Config.get("ape.mopDataPath", null);
```

**Widget ID matching** (important):
- rvsmart static data: `idName = "btn_encrypt"` (short form, no package prefix)
- APE `GUITreeNode.resourceId`: `"com.example.app:id/btn_encrypt"` (full form)
- Transform: `resourceId.contains(":id/") ? resourceId.split(":id/")[1] : resourceId`
- Handle null resourceId → fall back to activity-level score

**APE advantage over rvsmart**: `action.getResolvedNode()` gives direct `GUITreeNode` → no coordinate-based widget lookup needed.

#### 3b. MOP-Guided Action Scoring — `MopScorer.java`

**Injection point**: `StatefulAgent.adjustActionsByGUITree()` in `StatefulAgent.java` (line ~1113), after `action.setPriority(priority)`:

```java
// INJECT after priority assignment:
if (mopData != null && action.requireTarget() && action.isValid()) {
    GUITreeNode node = action.getResolvedNode();
    String activity = newState.getActivity();   // use newState.getActivity() — confirmed in State.java:222
    String shortId = extractShortResourceId(node.resourceId);
    int mopBoost = MopScorer.score(activity, shortId, mopData);
    action.setPriority(action.getPriority() + mopBoost);
}
```

**Scoring** (ported from rvsmart MopScorer):
- Direct MOP-reachable widget: `+500`
- Transitive MOP-reachable widget: `+300`
- Widget on MOP activity (no specific widget match): `+100`
- No match: `0`

**Files**: new `ape/utils/MopScorer.java`, modify `ape/agent/StatefulAgent.java`

#### ~~3c. Coverage Monitoring~~ — REMOVED FROM SCOPE

Runtime MOP coverage monitoring via logcat is removed. Rationale:
- rv-android already has full logcat infrastructure in Python (`rv-android-core/util/android/logcat_manager.py`, `rv-coverage/parser/log/logcat_parser.py`)
- Coverage metrics should be collected by the Python layer (rv-platform/rv-experiment), not inside the Java APE process
- Reading logcat from within `app_process` requires `READ_LOGS` permission which may be denied in Android 8+ (SELinux)
- No online feedback loop is needed — scoring is purely static-analysis-driven

Coverage data will be collected by the existing rv-android Python infrastructure as with other tools.

### Phase 4 — aperv-tool Plugin
**SDD Track**: Fast-Forward
**Repository**: `rvsec/rv-android/modules/aperv-tool/` (new module)

Create the rv-tools plugin that wraps the enhanced APE JAR. Follows the exact pattern of `rvsmart-tool`.

**Module structure**:
```
aperv-tool/
├── pyproject.toml                         # uv workspace member; entry-point declaration (informational)
└── src/aperv_tool/
    └── tools/aperv/
        ├── __init__.py
        ├── tool.py                        ← ApeRVTool(AbstractTool)
        ├── ape-rv.jar                     ← copied by Maven install (gitignored)
        └── .gitignore                     ← /ape-rv.jar
```

**ApeRVTool responsibilities**:
- Locate `ape-rv.jar` via `JarResolver` (priority: module dir → `$RVSEC_HOME/ape/target/` → `$TOOLS_DIR/aperv/`)
- Push JAR to device via ADB (`/data/local/tmp/ape-rv.jar`)
- Push static analysis JSON if `mopDataPath` variant requires it
- Execute via `app_process` with appropriate flags
- Extract metrics from stdout (activity coverage, state count)

**Registration** — add to `rv-platform/src/rv_platform/__init__.py` in `_register_external_tools()`:
```python
if not registry.is_tool_registered("aperv"):
    try:
        from aperv_tool.tools.aperv.tool import ApeRVTool
        registry.register_tool_class(ApeRVTool)
    except ImportError as e:
        logging.getLogger(__name__).warning(f"aperv tool not available: {e}")
    except Exception as e:
        logging.getLogger(__name__).error(f"Failed to register aperv tool: {e}")
```

**Variants**:
| Variant | Strategy | MOP data |
|---|---|---|
| `default` | sata | no |
| `sata` | sata | no |
| `sata_mop` | sata | yes — requires `static_data_path` (optional phase 3) |
| `bfs` | bfs | no |
| `random` | random | no |

**uv workspace**: `aperv-tool` is auto-discovered by `members = ["modules/*"]`. Add to `[tool.uv.sources]` in root `pyproject.toml` and declare as dependency in `rv-platform/pyproject.toml` (same pattern as rvsmart-tool).

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| `framework/classes-full-debug.jar` missing @hide APIs in some Android versions | Keep both compile-time JARs; do not replace with public android.jar |
| d8 rejects Java 11 bytecode | d8 supports Java 8–17; Java 11 is explicitly safe |
| Maven exec-plugin + d8 produces invalid classes.dex | Validate with `file -` and test `app_process` invocation before proceeding |
| Name conflict between `ape` builtin and `aperv` plugin in rv-tools | Distinct plugin IDs guarantee separation; `ToolRegistry` uses ID as key |
| MOP static data schema changes between rv-android versions | Parse only the minimum subset with graceful fallback when key is absent; phase is optional |
| `KEYCODE_MENU` ineffective on devices without hardware MENU button | Expected — on such devices MODEL_MENU will be tried once and become visited; low waste |
| MODEL_MENU ordinal insertion breaks ActionType.requireTarget() range | Verify ordinal range after insertion; MODEL_MENU must fall outside MODEL_CLICK..MODEL_SCROLL_RIGHT_LEFT range |
