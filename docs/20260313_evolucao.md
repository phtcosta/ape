# Análise Profunda: APE-RV — Limitações, Compatibilidade e Evolução

**Data:** 2026-03-13
**Contexto:** Análise realizada após conclusão das Fases 1–4 do APE-RV, antes do experimento comparativo (Fase 5).

---

## Sumário Executivo

- **APE-RV funciona em Android 6.0–10 (API 23–29)** — testado upstream; Android 11 provável mas não validado; **Android 12+ tem risco real** (Phantom Process Killer + @hide API changes).
- **Limitações críticas:** OOM em sessões longas, cegueira total a WebViews, text input aleatório sem semântica.
- **Evolução viável:** Android 12+ compat (baixo esforço), text input via LLM (médio esforço), stall recovery (médio esforço), wire ApeAgent/CEGAR (baixo esforço — código já existe).
- **APKs antigos continuam funcionando** após qualquer evolução do APE-RV, pois a ferramenta nunca é instrumentada no APK testado — só o Android OS do emulador importa.
- **Diferencial único do APE-RV:** MOP-guided scoring (Fase 3) — nenhum outro tool na literatura implementa isso.

### 1.2 APIs @hide Usadas pelo APE-RV

APE-RV acessa internals do Android que não fazem parte da SDK pública. São compilados via `framework/classes-full-debug.jar` (scope system — não empacotado no JAR final):

| API Interna | Arquivo | Risco de Quebra |
|-------------|---------|-----------------|
| `android.os.ServiceManager` | `AndroidDevice.java:51,84,89,94,115` | Alto (removido/restrito progressivamente) |
| `com.android.internal.view.IInputMethodManager` | `AndroidDevice.java`, `ApeAPIAdapter.java` | Médio |
| `android.hardware.display.DisplayManagerGlobal` | `AndroidDevice.java` | Médio |
| `android.app.ActivityManagerNative` | `ApeAPIAdapter.java` | Alto (removido no Android 10) |
| `android.app.IActivityManager` | `AndroidDevice.java` | Médio |
| `android.view.IWindowManager` | `AndroidDevice.java` | Médio |
| `android.content.pm.IPackageManager` | `AndroidDevice.java` | Médio |
| `com.android.internal.statusbar.IStatusBarService` | `AndroidDevice.java` | Baixo |
| `android.app.admin.IDevicePolicyManager` | `AndroidDevice.java` | Baixo |

**Mitigação existente:** `ApeAPIAdapter.java` tem 5 shims de reflexão que tentam múltiplas assinaturas de método com fallback gracioso. `ActivityManagerNative.getDefault()` já foi substituído por `ActivityManager.getService()` via reflexão.

**Gap:** Nenhum shim para os métodos de kill de processo (`iActivityManager.killPids()`). Se a assinatura mudar no Android 12+, falha silenciosa.

### 1.3 Phantom Process Killer (Android 12+, API 31+)

Android 12 introduziu um mecanismo que mata processos filhos quando usam CPU excessiva ou quando o pai é backgrounded. APE roda como processo filho via `app_process` — exatamente o padrão afetado.

**Risco:** Sessões de exploração longas (30+ minutos) em Android 12+ podem ser terminadas inesperadamente pelo sistema.

**Mitigação (requer ADB):**
```bash
adb shell device_config set_sync_disabled_for_tests persistent
adb shell device_config put activity_manager max_phantom_processes 2147483647
adb shell settings put global settings_enable_monitor_phantom_procs false
```
Android 14+ tem toggle em Developer Settings. Estas flags deveriam ser adicionadas a `scripts/run_emulator.sh` se o AVD for Android 12+.

**AVD RVSec atual:** Android 9 (API 28) — Phantom Process Killer não é problema. Mas qualquer atualização do AVD para Android 12+ requer mitigação.

### 1.4 APKs Antigos Continuam Funcionando?

**Sim, sem restrições** — independente de qualquer evolução que se faça no APE-RV.

**Raciocínio:** APE-RV nunca é instrumentado no APK testado. Ele roda como processo separado via `app_process` e interage com o app através da Accessibility API (AccessibilityNodeInfo). O APK testado permanece intocado.

A compatibilidade com APKs não depende do `targetSdkVersion` do APK, mas sim da **versão do Android OS do emulador/dispositivo**. Um APK com `targetSdkVersion=21` (Android 5.0) rodando num emulador Android 9 é perfeitamente testável pelo APE-RV.

**Implicação:** Evoluções como LLM text input, stall recovery, ou wire ApeAgent não têm impacto algum nos APKs testados. O único risco de "quebrar APKs antigos" seria uma mudança no APK sob teste (recompilação com nova instrumentação) — o que nunca é necessário para usar APE-RV.

## 2. Limitações Algorítmicas

### 2.1 Por Severidade

| Limitação | Severidade | Causa Raiz | Mitigação Atual |
|-----------|-----------|------------|-----------------|
| **OOM em sessões longas** | 🔴 Crítico | `Model.actionHistory` (ArrayList não limitado) + `State.treeHistory` acumulam sem limpeza. TODO explícito em `Model.java:124` | `flushImagesThreshold=10` (só screenshots, não GUITrees) |
| **WebView opaco** | 🔴 Crítico | Accessibility API não enxerga conteúdo web; `GUITreeNode.isWebView()` apenas identifica, não explora | `ignoreWebViewThreshold=64` (heurística fraca) |
| **Text input aleatório** | 🟠 Alto | `StringCache.nextString()` — sem semantic awareness, sem form filling, sem constraints. `truncateTextLength=8` | Nenhuma |
| **Fling ausente** | 🟠 Alto | TODO explícito `MonkeySourceRandom.java:256-260` marcado como "importante" mas não implementado | Apenas via fuzzing |
| **RecyclerView scroll** | 🟡 Médio | Excluído explicitamente — impossível inferir direção de scroll do class name sem LayoutManager | Documentado no código |
| **Pinch-zoom não model-guided** | 🟡 Médio | Só disponível via `doFuzzing` (2% por step) | Config `--pct-pinchzoom` |
| **CEGAR restrito** | 🟡 Médio | Refinamento só dispara em `NEW_ACTION_TARGET`; não detecta não-determinismo por estado interno do app | `actionRefinementFirst=true` |
| **State explosion cap** | 🟡 Médio | `maxStatesPerActivity=10` — satura prematuramente em apps complexos | Configurável |
| **Epsilon estático** | 🟡 Médio | `defaultEpsilon=0.05` fixo; sem decay schedule nem adaptação por confiança | Configurável |
| **Dialog detection heurística** | 🟡 Médio | `isTrivialActivity()` usa limiar de 5 arestas — frágil e app-dependente | Configurável |

| Uso em produção | Pesquisa | ByteDance CI | Pesquisa | Pesquisa |

**APE original (ICSE 2019):** Amplamente citado (Themis, FastBot2, GPTDroid, LLMDroid, etc.). É o baseline referencial da área.

### 3.2 Fraquezas Documentadas pelo Benchmark Themis (FSE 2021)

O Themis benchmark (52 bugs reais de apps open-source, 6 tools comparados) identificou para APE:
1. **Text input blindness** — bugs em login forms, search, validação de campos não são encontrados
2. **WebView dead zones** — apps hybrid têm grandes áreas inacessíveis ao modelo
3. **Model coarsening preference** — APE prefere coarsening para evitar state explosion → estados semanticamente distintos são fundidos → paths divergentes são perdidos
4. **18 de 52 bugs** não foram encontrados por nenhuma ferramenta — indica ceiling do paradigma atual

### 3.3 Proposta de Valor Única do APE-RV

APE-RV com MOP scoring (Fase 3) é o único tool que **direciona a exploração GUI para behaviors monitorados por propriedades de runtime (MOP)**. A questão de pesquisa não é "APE-RV tem mais cobertura que FastBot2?" — é "MOP scoring aumenta a cobertura de behaviors especificados em comparação com exploração sem guia?".

Esta é a Fase 5 (experimento comparativo): `sata` vs `sata_mop` em apps com instrumentação MOP.

## 4. Caminhos de Evolução

### 4.1 Priorizado por Esforço/Impacto

| Evolução | Esforço | Impacto | Notas |
|----------|---------|---------|-------|
| **Android 12+ compat** (Phantom Process Killer mitigation) | Muito Baixo | Alto para AVDs modernos | Adicionar flags `device_config` em `scripts/run_emulator.sh` |
| **Wire ApeAgent (CEGAR completo)** | Baixo | Médio | Classe `ApeAgent` existe mas não está conectada ao CLI `--ape ape`. Phase 2 deixou para depois. |
| **Semantic text input via LLM** (estilo QTypist, ICSE 2023) | Médio | Alto | Quando EditText encontrado, chamar endpoint LLM local (Ollama) para gerar input plausível. Config flag `ape.llmTextInput=true`. Ponto de injeção: `SataAgent.adjustActionsByGUITree()` |
| **LLMDroid-style stall recovery** | Médio | Alto | Quando `Model` não cresce por N steps, invocar LLM para sugerir próxima ação. +26% coverage no paper (FSE 2025). Ponto de injeção: `StatefulAgent` — mesmo padrão do MOP scoring |
| **Audit @hide APIs para Android 12–14** | Médio | Necessário para AVDs modernos | Verificar `ApeAPIAdapter.java` + `AndroidDevice.java` contra lista de hidden APIs restringidas nas APIs 31–34 |
| **FastBot2-style RL transition reuse** | Médio-Alto | Alto | Persistir modelo entre runs; reusar probabilidades de transição de experimentos anteriores. Extensão natural da serialização `sataModel.obj` existente |
| **TimeMachine-style snapshot rewind** | Alto | Alto | Integração com API de snapshots QEMU via ADB. Requer integração com emulador — arquiteturalmente ortogonal ao APE-RV atual |
| **Reescrita como LLM agent** (estilo DroidAgent/LELANTE) | Muito Alto | Substitui | Mudança de paradigma completa — não é evolução, é substituição |

### 4.2 Sequência Recomendada para Pesquisa RVSEC

1. **Agora:** Executar Fase 5 (experimento comparativo `sata` vs `sata_mop`) com AVD Android 9 atual — sem mudanças necessárias
2. **Próximo passo imediato:** Android 12+ compat (se houver planos de atualizar o AVD)
3. **Curto prazo:** Wire ApeAgent (CEGAR) — baixo esforço, habilita mais variantes experimentais
4. **Médio prazo:** Semantic text input via LLM — maior impacto em bug-finding em apps reais

## Apêndice A: Ferramentas da Literatura — Explicação Detalhada

### A.1 FastBot2 (ASE 2022 — ByteDance)

**Paper:** "FastBot2: Reusable Automated Model-based GUI Testing of Android Apps" — ASE 2022
**Origem:** ByteDance (empresa do TikTok/Douyin). Usado em produção no CI da empresa há 2+ anos.

**Ideia central:** Reinforcement Learning com reutilização de conhecimento entre execuções.
FastBot2 constrói um modelo de transições GUI (estado → ação → estado), mas ao contrário do APE, **persiste esse modelo entre runs** e usa probabilidades aprendidas para guiar futuras explorações. É essencialmente RL tabulado com replay de experiência.

**Arquitetura:**
- Roda via Accessibility Service + InstrumentationTest (não via `app_process`) → imune ao Phantom Process Killer
- Modelo probabilístico de transições: `P(s' | s, a)` atualizado incrementalmente
- Estratégia epsilon-greedy com decay adaptativo baseado em "novidade" do estado
- Suporte a múltiplos dispositivos em paralelo

**Resultados (Douyin/Toutiao, ~1B usuários):**
- 50.8% dos bugs reportados por developers foram detectados primeiro pelo FastBot2
- Supera APE, Monkey, Stoat em cobertura de código em apps complexos
- Detecta crashes de produção não encontrados por testes manuais

**Fraquezas:**
- Sem text input semântico (mesmo problema do APE)
- Cego a WebViews
- Persiste conhecimento de estrutura de UI, mas não de semântica de negócio

**Relevância para APE-RV:** O padrão de persistir `sataModel.obj` já existe no APE-RV — FastBot2 mostra como estender isso com reuso de probabilidades entre experimentos. Seria uma extensão natural.

### A.2 TimeMachine (ICSE 2020)

**Paper:** "Time-travel Testing of Android Apps" — ICSE 2020
**Origem:** Singapore Management University + Monash University

**Ideia central:** Snapshots de VM para eliminar o problema de "ficar preso".
Em vez de construir um modelo GUI, TimeMachine salva **snapshots do estado completo do emulador** (via QEMU) em momentos de "alta taxa de crescimento de cobertura". Quando a exploração estagna, **rebobina para um snapshot anterior** e recomeça de um ponto diferente.

**Arquitetura:**
- Wrappa o Monkey como backend de eventos
- Monitor de cobertura em tempo real (instrumentation de bytecode)
- Selector de snapshot: salva quando Δcoverage > threshold; rebobina quando cobertura para
- Requer emulador QEMU (não funciona em dispositivos físicos)

**Resultados (Themis + benchmarks próprios):**
- Cobertura mais alta de todos os tools no Themis benchmark (54% stmt avg)
- Encontra mais crashes únicos que APE, Monkey, Humanoid em experimentos controlados
- Custo: cada snapshot ocupa ~1-2GB de disco

**Fraquezas:**
- Sem text input semântico
- Depende de instrumentação de bytecode para medir cobertura — overhead ~20%
- Não funciona sem emulador QEMU com suporte a snapshot
- Não direcionado: rebobinar é aleatório, não orientado a objetivos

**Relevância para APE-RV:** A ideia de snapshot rewind seria a maior melhoria possível em cobertura, mas requer integração com QEMU — arquiteturalmente complexo. Não se integra naturalmente com o modelo do APE.

### A.3 GPTDroid (ICSE 2024) e LLMDroid (FSE 2025)

**Papers:**
- "Make LLM a Testing Expert: Bringing Human-like Interaction to Mobile GUI Testing" — ICSE 2024
- "LLMDroid: Enhancing Mobile App GUI Testing Coverage Guided by LLM" — FSE 2025

#### GPTDroid (ICSE 2024)

**Ideia central:** Reformula GUI testing como Q&A iterativo com um LLM.
A cada passo, GPTDroid envia ao GPT-4 uma descrição textual da UI atual (widgets, texto, hierarquia) e pergunta "qual ação executar para maximizar a descoberta de funcionalidades?". O LLM responde com a ação + raciocínio.

**Inovação: "Functionality-aware memory prompting":**
- Mantém um histórico comprimido das funcionalidades já descobertas
- Injeta esse contexto no prompt para evitar loops
- Resultado: o LLM "lembra" que já testou o login e foca em outras funções

**Capacidades únicas:**
- Text input semântico: "preencha um email válido para este campo" → LLM gera `user@example.com`
- Login flows: LLM entende o contexto e preenche credenciais de teste
- Search queries: LLM gera queries plausíveis para o domínio do app
- Form validation: LLM detecta campos obrigatórios e preenche coerentemente

**Fraquezas:**
- Latência alta: cada passo requer uma chamada GPT-4 (~1-3s) — 10-30x mais lento que APE
- Custo: ~$0.50/hora com GPT-4o (aceitável para pesquisa, problemático para CI/CD)
- Dependência de API externa (privacy concern com apps de empresa)
- Sem integração com MOP/RV

#### LLMDroid (FSE 2025)

**Ideia central:** Híbrido — usa tool existente normalmente, chama LLM só quando estagna.
"Quando a cobertura para de crescer por N steps consecutivos, o LLM analisa o estado atual e sugere novos pontos de entrada não explorados."

**Arquitetura:**
- Wrapper em torno de qualquer tool base (testado com Monkey e APE-like)
- Monitor de cobertura: detecta estagnação em janela deslizante
- LLM call: envia screenshot + hierarquia de UI → LLM sugere "você não explorou a aba Configurações"
- Executor: navega para o ponto sugerido e retoma exploração normal

**Resultados:**
- +26.16% cobertura de código vs baseline
- +29.31% cobertura de atividades
- Custo médio: $0.49/hora com GPT-4o
- 88% das sugestões do LLM levam a estados novos

**Por que é o caminho mais natural para APE-RV:**
1. Não substitui o APE — wrappa ou modifica minimamente
2. O ponto de injeção é `StatefulAgent` — o mesmo onde está o MOP scoring
3. Mantém o diferencial do MOP-guidance enquanto adiciona recuperação de estagnação
4. Custo controlável: LLM só é chamado quando necessário (não a cada step)
5. Pode usar LLM local (Ollama + Llama 3.1) para zero custo e privacidade total
